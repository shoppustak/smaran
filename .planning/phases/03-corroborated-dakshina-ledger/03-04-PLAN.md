## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| WhatsApp inbound webhook -> purohit-supplied free text | The dakshina amount is entered by the purohit as an unstructured WhatsApp text reply; the app must interpret it as trusted-but-unvalidated numeric input |
| Ledger row -> UPI deep link | The deep link is built from the purohit's own stored upiId; a malformed/unvalidated VPA would produce a payment link pointing nowhere useful |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Tampering | Free-text amount-capture handler in whatsapp.ts | mitigate | `findAwaitingAmountEntry` scopes the lookup strictly to the sender's own resolved `purohit.id`; the parsed amount is validated as a finite positive number before `recordDakshinaAmount` is called, and `recordDakshinaAmount` re-verifies `ledger.purohitId === purohitId` server-side before writing |
| T-03-08 | Information Disclosure | UPI deep link dispatch (buildUpiDeepLink call site in whatsapp.ts) | mitigate | `isValidUpiId(purohit.upiId)` is checked immediately before building the deep link; on failure the flow fails closed (alert sent, no card dispatched) per CONTEXT.md D-02, rather than sending a broken/misleading payment link |
| T-03-09 | Repudiation | Duplicate "ritual completed" taps | mitigate | The `ritual-completed` handler checks for an existing ledger row by `eventId` + `purohitId` before creating one; repeat taps either resend the amount-request prompt or no-op, never create a second ledger row for the same event |
</threat_model>
