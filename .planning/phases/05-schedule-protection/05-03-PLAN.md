Code smells (fixed opportunistically): `getMuhuratCollision` fetches ALL of a purohit's events

`parseGregorianHint`'s silent today/09:00 fallback hides missing data instead of surfacing it.

## Design note — why collision needs no Gregorian resolution
Two annual rituals collide iff they share the same (maas, paksha, tithi) — which is the same
Gregorian day in any given cycle by definition — AND the same muhurat window. So the correct
collision predicate is a tuple match on (maas, paksha, tithi, window); it needs no panchang call
and is immune to ingest-timing. Only the day-sheet ("my week" = the next 7 *real* days) needs
Gregorian dates, and `resolveUpcomingEventsForWeek` (brain.ts:162) already produces them.
Edge cases (kshaya/adhika tithi spanning or skipping a Gregorian day) are acceptable for a
collision *warning*; they were already unhandled and are strictly better than the today-default.
