## Design note — activation without a payment webhook
The product forbids PSP/webhook-driven payment state and models money events as two-sided button
taps (see the Phase 3 ledger: pending → claimed → corroborated). A subscription must follow the
same philosophy. The ₹29/mo UPI Autopay mandate link is a DEEP LINK the family opens in their own
UPI app to set up recurring payment directly against the purohit's VPA — the platform never sees the
money and cannot (per constraint) consume a payment webhook to learn it landed. So activation state
must be **self-asserted by a button tap**, not pushed by an inbound webhook: family taps the offer's
CTA URL (opens UPI app) → taps a follow-up "मैंने सदस्यता ले ली" confirm reply → an
ownership-checked handler moves `family_sub_status` none→active and stamps `family_sub_renews_at`.
Renewal/lapse is date-driven by the scheduled sweep, and the nudge that re-engages a lapsing family
is a registered Utility template. This keeps the phase strictly inside "no webhook-driven payment
state" while still delivering a tracked, renewable subscription. The unauthenticated mandate webhook
is deleted, not patched — an auth header on a forbidden pattern is still the forbidden pattern.
</audit_findings>
