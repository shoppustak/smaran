## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Purohit free-text answers → onboarding_state / purohits columns | Every captured field originates as untrusted WhatsApp message text |
| onboarding.ts → internal `/api/panchang` self-call | Same-process HTTP call, no network boundary crossed, but still wrapped defensively |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02 | Tampering | Every per-step field capture (name/city/upi) | mitigate | Every captured field is `.trim()`-ed and length-capped (200 chars for name/city, 256 for UPI ID) before being written to `onboarding_state`/`purohits`; empty or oversized input is rejected with a re-prompt at the same step, never silently truncated or passed through unchecked. |
| T-01-05 | Tampering / Injection | `db.insert`/`.update`/`.delete` calls in this file | mitigate | All writes go through Drizzle's parameterized query builder (`.values()`, `.set()`, `eq()`) — no raw SQL string is constructed from purohit-supplied text anywhere in this file. |
| T-01-09 | Repudiation / DoS (partial) | Internal `/api/panchang` self-fetch on the completion path | accept | Single-attempt, try/catch-wrapped, falls back to a degraded (but still 2-element) reply on failure — never blocks or retries indefinitely. Since this is a same-process localhost call, not a network trust boundary, risk is limited to the Vedika sandbox's own availability, already accepted for `panchang.ts` itself. |
</threat_model>
