<objective>
Build the resumable onboarding conversation state machine: `handleOnboardingMessage(phoneNumber,
text)` walks a purohit through name ‚Üí city ‚Üí ward ‚Üí UPI ID ‚Üí calendar system (D-03: resumable by
`phone_number`, persisted ‚Äî never held only in WhatsApp/server memory), then on completion
persists the real `purohits` row and composes the wow-moment card (D-02: canned example family,
real Vedika-resolved date).

Purpose: This is the core business-logic artifact for both of this phase's requirements (ONBD-01
capture, ONBD-02 wow card). It has zero HTTP/route surface of its own ‚Äî `01-04-PLAN.md` wires it
into the webhook handler (interface-first ordering).
Output: `code/artifacts/api-server/src/lib/onboarding.ts`, fully implementing the 5-step flow +
completion routine.

**Revision note (targeted fix, follow-up to `01-02-PLAN.md`'s revision):** `geocodeCity`'s
signature changed from `geocodeCity(city)` to `geocodeCity(city, ward)` ‚Äî `ward` required ‚Äî because
Nominatim returns no sub-city field for a bare city-name query, so ward-level disambiguation must
come from the purohit's own answer (see `01-02-PLAN.md`'s Task 1 revision note for the live
verification). `01-02-PLAN.md` left the choice of how to capture `ward` to this plan, between (a) a
new dedicated "ward" step in the sequence, or (b) folding it into the existing "city" step as one
compound free-text answer ("city and area") parsed before calling `geocodeCity`.

**Decision: (a) a new dedicated "ward" step, inserted between "city" and "upi".** Reasoning:
`onboarding_state.current_step` is a plain `text` column (`code/lib/db/src/schema/onboardingState.ts`,
`.default("name")`), not a Postgres enum ‚Äî adding a new step value is a TypeScript-only change (one
new entry in the `ONBOARDING_STEPS` tuple/type union, one new `switch`/`if` branch) with **zero DB
migration**. The resume-by-phone-number logic already branches generically on
`draft[0].currentStep`'s string value, so a 5th step slots in without touching that mechanism.
`onboarding_state` also has no spare column to hold a raw "ward" substring anyway ‚Äî this plan avoids
needing one by consuming the ward answer immediately in the same request that computes it: the
"city" step now stores raw city text only (no geocode call) and advances to "ward"; the "ward" step
reads the already-persisted `city` from the draft row, geocodes `(city, wardAnswer)` in that same
turn, and only the *result* (`latitude`/`longitude`/`localityKey`) is persisted ‚Äî matching columns
that already exist. Option (b) was rejected: splitting one free-text answer like "Mumbai Andheri
West" or "Andheri West, Mumbai" into city/ward parts is inherently ambiguous (no reliable delimiter
convention across purohits), and this whole revision exists to make `locality_key` disambiguation
*more* reliable, not to add parsing risk on top of it. A dedicated step costs one extra WhatsApp
round-trip but is deterministic ‚Äî worth it here.
</objective>
