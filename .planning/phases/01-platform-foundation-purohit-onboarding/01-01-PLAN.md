<design_note>
## Design Note: why a 5th table beyond the fixed schema

The fixed `purohits` SQL schema declares `name`, `city`, `latitude`, `longitude`, `locality_key`,
`upi_id` all `NOT NULL`. CONTEXT.md's D-03 requires onboarding state to be resumable and persisted
(not held only in WhatsApp/server memory), keyed by `phone_number`. A purohit who has only answered
"name" so far cannot be represented as a row in `purohits` without violating those NOT NULL
constraints. CONTEXT.md's "Claude's Discretion" section explicitly leaves this representation choice
to the planner: "a `status`/`current_step` column on `purohits` vs. a separate onboarding-state
table." This plan chooses the **separate table** (`onboarding_state`) so the fixed `purohits` schema
stays byte-for-byte identical to `.planning/intel/constraints.md` тАФ no extra columns bolted onto a
locked schema. `onboarding_state` is purely internal/operational, never referenced by
`.planning/intel/constraints.md`'s 4-table contract, and gets deleted once a real `purohits` row is
inserted (see `01-03-PLAN.md`).
</design_note>
