**Revision note (this plan only, targeted fix):** An external review of Phase 2 planning found
that `geocodeCity`'s original single-argument (`city`-only) form produces an identical
`locality_key` for every purohit onboarded in the same city, because Nominatim has no
ward/temple-cluster granularity for a bare city-name query — and Phase 2's density-accounting
work (§7) stamps that same key onto every subsequent `yajmans`/`events` write, so the
degradation compounds and can't be repaired retroactively for the seed cohort once written. This
revision changes `geocodeCity`'s signature to `geocodeCity(city, ward)` (ward now required) and
folds the ward into `localityKey`'s derivation. See Task 1 for the live verification that ruled
out reusing Nominatim's own address breakdown instead of asking the purohit. **Downstream
impact (not fixed by this plan — flagged for follow-up):** `01-03-PLAN.md`'s "city" step

has no ward-capture step; `01-04-PLAN.md`'s E2E flow correspondingly only ever posts a city
answer. Both will fail typecheck / no longer reflect the real flow until revised to capture a
ward/area answer (either as its own step, or folded into a single "city and area" free-text
answer at the existing city step) and thread it into `geocodeCity(city, ward)`. This plan
deliberately does not make that change — it is out of scope per this revision's instructions.

    to come from the purohit, not be inferred from Nominatim's response. This keeps D-05
    (Nominatim as provider) intact — it changes what's asked alongside the city, not the provider.

    Create `code/artifacts/api-server/src/lib/geocoding.ts` exporting
    `export async function geocodeCity(city: string, ward: string): Promise<{ latitude: number; longitude: number; localityKey: string } | null>`.
    `ward` is a purohit-supplied free-text ward/area/neighbourhood name (e.g. "Andheri West",
    "Sadar Bazaar") — required, not optional, so the caller (`01-03-PLAN.md`'s state machine,
    revised separately) cannot silently skip capturing it.

    Base URL: `const NOMINATIM_BASE_URL = process.env.NOMINATIM_BASE_URL ?? "https://nominatim.openstreetmap.org";`
    (unlike Vedika, Nominatim needs no API key and no sandbox/production split — this is simpler
    than `panchang.ts`'s env-var flip, just one optional override for testability).

    Factor the single-result GET into a private helper, e.g. `async function
    searchNominatim(query: string): Promise<{ lat: string; lon: string } | null>`, so there is
    exactly one `fetch(` call site in the file even though it may be invoked twice per
    `geocodeCity` call:
    `GET ${NOMINATIM_BASE_URL}/search?format=json&limit=1&q=${encodeURIComponent(query)}` with
    header `"User-Agent": "Smaran/1.0 (contact: shop.pustak@gmail.com)"` — Nominatim's usage
    policy requires a descriptive identifying User-Agent. On non-`ok` status or an empty result
    array, `searchNominatim` returns `null` (it does not itself log or throw — that's
    `geocodeCity`'s job, so the two-attempt logic below stays simple).

    In `geocodeCity`: trim both `city` and `ward`. Wrap the whole body in one `try`/`catch`
    (network errors caught once, at the outer level — never throw, return `null`). Attempt 1:
    call `searchNominatim(`${trimmedWard}, ${trimmedCity}`)` (ward-qualified — best chance of a
    precise match if OSM has that locality mapped). If that returns `null` (empty result, not a
    network error — a thrown error propagates to the outer `catch` and skips attempt 2 entirely,
    since a network failure on attempt 1 means attempt 2 would fail identically), attempt 2:
    call `searchNominatim(trimmedCity)` (city-only fallback — keeps geocoding successful even
    when the specific ward isn't a mapped OSM place; the ward is still captured in `localityKey`
    regardless of whether Nominatim could resolve it as a place). If both attempts return `null`,
    `logger.warn({ city, ward }, "Nominatim geocode failed")` and return `null`.

    On a successful match from either attempt: Nominatim returns `lat`/`lon` as **strings**,
    coerce with `Number()`. Derive `localityKey` as a normalized slug combining city AND ward —
    **not** from the Nominatim response (confirmed above: no reliable sub-city field exists for a
    city-only query, and attempt 1's ward-qualified query may still fail to resolve as a place
    even though the purohit's ward text itself is valid input) — using a shared slug helper:
    `const slugifyPart = (s: string) => s.trim().toLowerCase().replace(/[^a-z0-9]+/g,
    "-").replace(/(^-|-$)/g, "");` then `const localityKey = trimmedWard.length > 0 ?
    `${slugifyPart(city)}-${slugifyPart(ward)}` : slugifyPart(city);` (the `trimmedWard.length >
    0` branch is defensive only — the caller is expected to always supply a non-empty ward per
    the required-parameter contract above). Document this derivation choice with an inline code
    comment since it's a planner-level discretionary resolution, not something specified in
    CONTEXT.md. Keep the combined key coarse (city+ward), not venue/GPS-precise — matches
    01-DISCUSSION-LOG.md's original "coarse... not precise venue matching" intent for
    `locality_key`; do not attempt to geocode down to street/venue level.

    Import `{ logger }` from `"./logger"` (module-level pino instance, not `req.log` — this
    function isn't bound to an Express request).
