## Trust Boundaries

|----------|--------------|
| Meta -> POST /api/whatsapp/webhook | Untrusted inbound HTTP from Meta's infrastructure; this endpoint now triggers real DB writes via the onboarding flow |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | POST /api/whatsapp/webhook | accept | No X-Hub-Signature-256 HMAC verification is implemented this phase -- WHATSAPP_APP_SECRET is not yet a provisioned/documented env var anywhere in this repo (.env.example, render.yaml, or STATE.md's tracked credential list), so adding verification here would require introducing an undocumented secret with no confirmed provisioning path (a legitimate "missing information" constraint, not a difficulty judgment). The test-mode webhook is already gated by an unguessable URL plus the WHATSAPP_VERIFY_TOKEN handshake, and this phase carries no payment or ledger data (UPI ID is format-checked only, never charged). Flag WHATSAPP_APP_SECRET signature verification as a follow-up for the staging-prod-infra hardening pass before scaling outbound. |
| T-01-04 | Information Disclosure | GET /api/purohits/:phoneNumber | mitigate | Requires a matching `X-Internal-Key` header against the `INTERNAL_API_KEY` env var; returns 401 if missing or mismatched, checked before the DB is ever queried. Not linked from anywhere purohit-facing -- inspection-only, used by the E2E suite and manual ops checks. |
</threat_model>
