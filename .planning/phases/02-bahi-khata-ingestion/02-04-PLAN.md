<interfaces>
Exact extraction JSON shape from §5.1 (system prompt must demand this exact shape, temperature 0):
```json
{
  "family_name": "string | null",
  "gotra": "string | null",
  "whatsapp_number": "string | null",
  "events": [{
    "event_type": "shraddh | katha | birthday | griha_pravesh | anniversary | other",
    "label": "string | null",
    "maas": "string | null",
    "paksha": "Shukla | Krishna | null",
    "tithi_name": "string | null",
    "gregorian_hint": "string | null"
  }],
  "confidence_notes": "string | null"
}
```
Rules embedded in the prompt (§5.1, verbatim): never guess a field not spoken; multiple events per
note are normal (pitaji + mataji in one breath); exactly one family per note — if two families are
detected, return the first and set `confidence_notes: "multi-family"`.

§10 photo-path delta: same call shape, image input instead of transcript text, but `events[]` may
span **multiple families per page** — the one structural difference. Cap at 5 families per page
burst; then "अगला पन्ना भेजें".

D-02 (locked decision, non-negotiable): "The extraction-model choice ... is not pre-committed — it
is decided empirically by extending §11's eval harness with a fourth column (extraction model
alongside ASR provider), bake-off candidates: Claude Haiku 4.5, Gemini Flash, and Sarvam-M.
Selection criterion is per-field accuracy (>=90% acceptance bar per §11), not preference." This
plan must NOT hardcode a production default among the three — see the ModelCaller design below.

ModelCaller design (this plan's own architecture, extending §4's adapter-boundary discipline to
extraction models, since §11 requires comparing 3 concrete candidates):
```typescript
interface ModelCaller {
  call(systemPrompt: string, userContent: string | { imageBytes: Buffer }): Promise<unknown>;
}
```
`extraction-models/index.ts`'s `getExtractionModelCaller(): ModelCaller` reads `EXTRACTION_MODEL`
env var (`"claude-haiku-4.5" | "gemini-flash" | "sarvam-m"`). If unset (no bake-off has run yet),
return a caller whose `.call()` immediately rejects with a clear `ExtractionModelNotConfiguredError`
("no EXTRACTION_MODEL selected — run the §11 eval harness bake-off (D-02) and set EXTRACTION_MODEL
+ the corresponding API key"). This is intentional: production `ingest.ts` (plan 02-07) will call
this and gracefully fail into `ingest_jobs.status = 'failed'` until a human runs the bake-off and
sets the env var — a known, spec-mandated sequencing dependency (§12's build order puts the bake-off
before the confirm-card/writes stages), not a bug.

External-API-proxy shape to mirror for each of the 3 concrete callers (`panchang.ts`, same shape
already used for `asr/sarvam.ts`/`asr/openai.ts` in plan 02-03): env-gated key, `fetch`, `if (!upstream.ok)`
throw a typed error, JSON body in/out.
