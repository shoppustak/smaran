<objective>
Add ops routes for `ingest_jobs`: a PII-minimal, `X-Internal-Key`-gated status-inspection GET
endpoint, and a `POST /ingest-jobs/purge` endpoint an external cron pinger hits daily to enforce
`docs/ref-state-2-plan.md` §8's retention table (raw transcript purged 30 days after a job reaches
a terminal status, extraction JSON purged on the same schedule) AND to expire stale
`awaiting_confirm` jobs that never reach a terminal status at all (F-6).

Purpose: §8's retention policy is a DPDP-alignment promise, not optional cleanup — "this must be
true, because the landing page and the sabha pitch will say it is." This plan is where that promise
becomes enforced code, following this repo's existing "$0-infra, external pinger hits a plain
endpoint" posture (the same shape `/keepalive` already established) rather than introducing a new
job-scheduler dependency. The GET endpoint additionally replicates Phase 1's `X-Internal-Key` gate
(F-5), closing an inconsistency between this phase's two ops-inspection endpoints and Phase 1's.
Output: 2 new OpenAPI paths + generated zod schemas, `routes/ingest-jobs.ts`, registered in
`routes/index.ts`.
</objective>

§8 retention table (exact):
| Artifact | Kept | Then |
|---|---|---|
| Audio bytes | Until transcription succeeds | Deleted immediately — never persisted (already satisfied structurally by plan 02-05's media.ts) |
| Raw transcript | 30 days (debugging window) | Purged by scheduled job |
| Extraction JSON | Until confirm/reject + 30 days | Purged |
| Confirmed fields | Permanent (the product) | Deletion-on-request honors cascade |

F-6 fix — stale `awaiting_confirm` expiry (extends §8, does not replace it): §8's table promises
transcript/extraction are purged "30 days, full stop" — a job that never reaches a terminal status
(`confirmed`/`rejected`/`failed`) because the purohit simply went quiet is NOT covered by the
terminal-status-only purge above, and D-03 explicitly defers any reminder nudge for such a job to
Phase 4 — meaning nothing else in this phase's lifecycle ever moves it out of `awaiting_confirm`. The
purge job must ALSO transition `status = 'awaiting_confirm'` rows older than 30 days (same window,
chosen because it matches §8's own stated debugging-window length and needs no separate policy to
justify) to `status = 'rejected'`, `error = 'expired'`, with `transcript`/`extraction` nulled the
same way the terminal-status path nulls them — this is a SEPARATE `UPDATE` statement in the same
purge run, not a modification of the terminal-status query's `WHERE` clause.
