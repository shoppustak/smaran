Important: `ingest_jobs.field_scores` (plan 02-07's `fieldScores`) is `Record<string, number>` —
match SCORES only, never the matched canonical string. `ingest_jobs.extraction` holds the raw
as-heard fields. `confirmJob` therefore cannot read a stored canonical value from either column — it
must re-derive canonical `maas`/`paksha`/`tithi_name`/`gotra` at confirm time by re-running
`matchField` against the raw `extraction` fields, the same way plan 02-07 does when it builds the
confirm card's `canonicalExtraction` (see plan 02-07 Task 2). This is a cheap, pure re-computation
(no I/O), not a second source of truth — and because both plans apply the identical
matchField+vocab+threshold logic to the same raw `extraction`, the canonical value `confirmJob`
writes is always the same value the confirm card displayed for that field (this keeps display and
write in agreement — see plan 02-07 Task 2 for the display-side half of this). `family_name` is the
one exception: it is never routed through `matchField` at all, by either plan (D-05) — it is not a
"re-derivation with a different threshold," it has no threshold, ever.

`maas` and `paksha` are TEXT columns on `events` (per `.planning/intel/constraints.md`'s base
schema) — the re-derived canonical STRING is what gets written for those two fields, with the
as-heard string as fallback when `matchField` returns `canonical: null` (never null the column).
`gotra` is a TEXT column on `yajmans` (nullable) — per D-05 it gets the SAME resolve-or-fallback
treatment as `maas`/`paksha` (re-run `matchField` against `gotraVocab` with `GOTRA_MAX_EDITS`,
write the resolved canonical string when matched, fall back to the as-heard `extraction.gotra`
string only when `matchField` returns `canonical: null` — gotra no-match is valid per §5.2, never
force to the nearest list entry, but a CONFIDENT match must be written, not discarded). `gotra`'s
threshold (`GOTRA_MAX_EDITS`, ≤1 edit) is intentionally tighter than `maas`/`paksha`/`tithi_name`'s
(≤2 edits) per D-05 — do not reuse `MAAS_MAX_EDITS` or any other field's threshold constant for
gotra. `family_name`, by contrast, gets NO `matchField` call at all — it is written exactly as
`extraction.family_name` (trimmed/whitespace-collapsed/NFC-normalized per the family-name
normalization step below, but never fuzzy-corrected).
`events.tithi`, however, is `SMALLINT NOT NULL` (1-15, per `.planning/intel/constraints.md`), NOT a
string column — writing the canonical `tithi_name` string (e.g. "Dwadashi") into it is a type
error / DB-layer rejection. `confirmJob` must take the extra step of looking up that resolved
canonical `tithi_name` in `tithiVocab` (plan 02-02's
`(VocabEntry & { tithiNumber: number; impliedPaksha?: ... })[]`) via
`tithiVocab.find(v => v.canonical === resolvedTithiName)?.tithiNumber` and insert THAT integer into
`events.tithi` — never the string. If that lookup comes back `undefined` (i.e. `matchField` itself
returned `canonical: null` for `tithi_name` — structurally unreachable in practice since the confirm
card's ❓-gating already forces the purohit through a canonical candidate pick or free-text re-entry
before ✓ is even offered, but `confirmJob` re-derives independently and defensively), do not insert
any events for this job — instead call `transitionJob(job.id, "failed", { error: "tithi_unresolved" })`
rather than inserting an undefined/NaN value into the NOT NULL integer column, AND send the purohit
the same D-04 friendly re-record/re-photograph apology message plan 02-07's guard-rail and
catch-all failure paths send via `sendWhatsappMessage` — every failure->failed transition in this
phase notifies the purohit, this one is no exception; a silent fail-closed here would leave them
tapping ✓ into the void with no signal at all.
