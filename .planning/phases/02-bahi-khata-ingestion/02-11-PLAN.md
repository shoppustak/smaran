<objective>

confirm card's fields, picking a field opens a WhatsApp interactive list of the 3 nearest canonical
candidates plus a "कुछ और" free-text fallback, and either path patches the job's draft fields and
re-sends a corrected confirm card — all before any write happens (writes remain exclusively
`confirmJob`'s job, per plan 02-08).

presents vision-extracted draft entries for correction and confirmation") both depend on this flow

list, 3 candidates, free-text fallback) is locked verbatim by CONTEXT.md — this plan is where that
lock becomes real code, reusing plan 02-06's `topCandidates` rather than re-deriving a second
ranking implementation.

Output: `confirm-card.ts` extended with the list-builders, `ingest.ts` extended with the
correction-state functions, `whatsapp.ts`'s interactive/text branches extended to route them,
`tests/api/ingest.spec.ts` extended with acceptance coverage for the new reply-id prefixes.

§6 exact correction rule (locked, do not reduce): "All fields >= threshold: card as above. `✓`
writes everything (Stage H). `✏` opens a numbered list — reply with the item number, then a
WhatsApp interactive list message offering the 3 nearest canonical candidates + "कुछ और" (free-text
re-entry for that field only)."

Reply-id grammar this plan introduces (extends plan 02-08's `{action}:{jobId}` convention with a
third and fourth segment; `fieldPath` itself never contains a colon, so parsing is a plain
`id.split(":")`):
- `edit:{jobId}` — ✏ tap (plan 02-08 already sends this id on the confirm card; this plan is the
  first to actually handle it)
- `field:{jobId}:{fieldPath}` — purohit picked a field from the numbered list
- `candidate:{jobId}:{fieldPath}:{index}` — purohit picked one of the 3 canonical candidates (`index` is `0`-`2`)
- `freetext:{jobId}:{fieldPath}` — purohit picked "कुछ और"; the NEXT plain-text message from that
  purohit is the replacement value for `fieldPath`

`fieldPath` grammar (this plan's own design — dots only, so it never collides with the colon-
delimited reply-id grammar above): `"family_name"`, `"gotra"`, or `"events.{i}.maas"` /
`"events.{i}.paksha"` / `"events.{i}.tithi_name"` for the `i`-th event in `extraction.events`.
`family_name` is EXCLUDED from candidate-list generation (D-05: never fuzzy-matched) — selecting it
routes straight to free-text.

`confirm-card.ts` entry points this plan consumes (plan 02-06 output — reuse, do not re-implement):
`topCandidates(heard: string, vocab: VocabEntry[], n: number): string[]`, the `ExtractionResult`
shape it renders from, and `sendWhatsappMessage`'s payload shape (`whatsapp-client.ts`, plan 02-05
output): `{ type: "interactive"; interactive: Record<string, unknown> }`.

`fuzzy-match.ts` / `vocab/*.ts` entry points (plan 02-02 output, same ones plan 02-08's `confirmJob`
already consumes): `MAAS_MAX_EDITS`/`PAKSHA_MAX_EDITS`/`TITHI_MAX_EDITS`/`GOTRA_MAX_EDITS`,
`maasVocab`/`pakshaVocab`/`tithiVocab`/`gotraVocab`.

`ingest.ts` entry points this plan extends/consumes (plans 02-07/02-08 output, same file):
`transitionJob(jobId, status, patch)`, `IngestJob` (has `id`, `purohitId`, `status`, `extraction`,
`fieldScores`), the DB-gating pattern and purohit-by-id lookup `confirmJob` already established in
plan 02-08.

Storage note: this plan does NOT change the `ingest_jobs` schema (plan 02-01, untouched). The
"awaiting free-text reply for field X" marker is carried as an extra key inside the existing
`field_scores` JSONB column (already a flexible JSON blob at the DB layer) — `_pendingCorrectionField:
string`, sitting alongside the per-field score numbers plan 02-07 writes there. This plan adds a new
`patchJob(jobId, patch)` helper in `ingest.ts` for field-only updates that do NOT move `status` (a
pure column patch, distinct from `transitionJob`'s forward-only status-transition contract) — do not
modify `transitionJob` itself.

`whatsapp.ts`'s interactive branch this plan extends (plan 02-08 Task 2 output — same file, same
branch, additional `if`/`else if` arms only): currently routes `confirm:*`/`reject:*`/
`collision-yes:*`/`collision-no:*`; any other prefix (today: `edit:*`/`field:*`/`candidate:*`/
`freetext:*`) falls through to a log-warning-and-no-op default. This plan replaces that fallthrough
for exactly those 4 prefixes with real handling, leaving the default no-op for anything still
unrecognized.

`whatsapp.ts`'s text-message branch this plan extends (see plan 02-08's interfaces block for the
current shape — a flat ring-buffer logger, no state machine today): add a check before/alongside the
existing logging — if the sender has a job with a pending free-text correction, treat this text as
the correction value instead of (or in addition to) logging it normally.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| purohit's correction reply (interactive list pick or free-text message) -> ingest.ts field patch -> re-rendered confirm card | untrusted input rewrites the draft record that confirmJob will later write verbatim |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-25 | Tampering | correction reply on a non-`awaiting_confirm` job | mitigate | Every correction function (startCorrection/selectCorrectionField/applyCorrectionCandidate/beginFreeTextCorrection/applyFreeTextCorrection) checks `status === "awaiting_confirm"` before touching the job, mirroring confirmJob's own idempotency guard — a correction reply for an already-confirmed/rejected/stale job is a no-op, not a stray patch |
| T-02-26 | Elevation of Privilege | `fieldPath` parsed from an inbound reply id | mitigate | `fieldPath` is validated against the fixed allowlist grammar (`family_name`/`gotra`/`events.{n}.(maas\|paksha\|tithi_name)`) before `getFieldValue`/`setFieldValue` ever run — a crafted reply id cannot write into an arbitrary object key |
| T-02-27 | Tampering | free-text correction bypasses fuzzy-match confidence gating | accept | The purohit explicitly typed the replacement after seeing the field flagged wrong — this is the same trust level §6 already extends to the ✓ tap itself; the field's score is set to `1` because a human, not the AI, now asserts the value |
</threat_model>
